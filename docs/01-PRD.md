## 1. 产品定位

**一句话定位**：VTCC 是面向 Valorant（无畏契约）电竞观众的 VCT 赛事信息查询工具。

VTCC 不是社区、不是资讯媒体、也不是数据分析平台，而是一个轻量级、高频查询的赛事信息工具，核心价值是「快、准、全」：

- **快**：3 秒内查到今天有什么比赛
- **准**：官方数据同步，比分、赛程准确
- **全**：赛事、战队、选手信息一站式查询

对标产品可理解为「电竞版体育资讯速查工具」，或类似 HLTV（CS 赛事站）在 Valorant 领域的小程序化实现。

### 核心用户价值

解决「VCT 比赛太多、赛区太多、时间不好记」的痛点，尤其是国际赛区比赛存在时差问题时，帮助用户快速定位赛程与结果。

## 2. 用户画像

| 画像 | 特征 | 核心诉求 |
|---|---|---|
| 硬核观众 | 每个赛区比赛都看，关注选手数据、战队打法 | 深度数据、选手履历、历史交锋 |
| 泛观众/路人粉 | 只关心热门赛事（如冠军赛、Masters） | 简单直接的赛程提醒、比分速览 |
| 玩家兼观众 | 自己玩游戏，也看职业比赛借鉴打法 | 战队/选手动态、比赛录像入口（后续） |
| 电竞爱好者/自媒体 | 需要快速查数据做内容 | 数据的可查、可比较、可引用 |

### 典型使用场景

- 通勤路上刷「今天有什么比赛」
- 比赛前查两支战队历史交锋
- 看比赛时查某个选手是谁、什么打法
- 关注的战队出现在赛事列表里，第一时间关注

## 3. 用户使用流程

**核心路径（高频）**

```
打开小程序 → 首页看"今日比赛" → 点击比赛 → 查看比赛详情（比分/阵容）
                                                ↓
                                      点击战队/选手 → 查看详情
```

**搜索路径**

```
首页搜索框 → 输入战队/选手/赛事名 → 搜索结果 → 进入对应详情页
```

**收藏路径**

```
战队/选手详情页 → 点击收藏 → 我的收藏列表 → 快速回访
```

**赛事浏览路径**

```
首页"热门赛事" 或 底部Tab"赛事" → 赛事列表 → 赛事详情/该赛事下比赛列表 → 比赛详情
```

## 5. 页面跳转关系

**首页 P01**
- 点击"今日比赛"卡片 → 比赛详情 P04
- 点击"热门赛事"卡片 → 赛事详情 P03
- 点击搜索框 → 搜索页 P07
- 底部 Tab 切换 → 赛事列表 P02 / 我的收藏 P08 / 我的 P09

**赛事列表 P02**
- 点击赛事卡片 → 赛事详情 P03

**赛事详情 P03**
- 点击赛程中的某场比赛 → 比赛详情 P04
- 点击参赛战队 → 战队详情 P05
- 点击收藏按钮 → 加入收藏（无跳转，Toast 提示）

**比赛详情 P04**
- 点击战队名/Logo → 战队详情 P05
- 点击选手名 → 选手详情 P06
- 点击所属赛事 → 赛事详情 P03

**战队详情 P05**
- 点击阵容中选手 → 选手详情 P06
- 点击近期比赛 → 比赛详情 P04
- 点击收藏按钮 → 加入收藏

**选手详情 P06**
- 点击所属战队 → 战队详情 P05
- 点击收藏按钮 → 加入收藏

**搜索页 P07**
- 结果为赛事 → 赛事详情 P03
- 结果为战队 → 战队详情 P05
- 结果为选手 → 选手详情 P06

**我的收藏 P08**
- 点击收藏的战队 → 战队详情 P05
- 点击收藏的选手 → 选手详情 P06
- 点击收藏的赛事 → 赛事详情 P03

> 设计原则：详情页之间可互相跳转（战队 ↔ 选手 ↔ 比赛 ↔ 赛事），形成信息网络而非层级树，方便用户顺藤摸瓜查询。

## 6. 每个页面需要展示的数据

### P01 首页
- 今日比赛列表：战队A vs 战队B、时间、赛事名、赛区标签、直播/已结束状态、比分（若已结束）
- 热门赛事卡片：赛事 Logo、赛事名、阶段、赛区、进行中标签
- 搜索入口：占位提示文案（如"搜索战队/选手/赛事"）

### P02 赛事列表页
- 赛事卡片：Logo、名称、赛区（美洲/EMEA/太平洋/中国等）、时间范围、状态（进行中/未开始/已结束）
- 筛选项：按赛区、按状态

### P03 赛事详情页
- 赛事基础信息：名称、Logo、赛区、时间范围、赛制说明
- 赛程/对阵表（按轮次/阶段分组）
- 参赛战队列表
- 收藏按钮

### P04 比赛详情页
- 对阵双方战队（Logo、名称）+ 比分
- 比赛时间、所属赛事、赛区、BO 几
- 分地图比分（Map 1/2/3 比分）
- 双方出场阵容（选手名单）
- 比赛状态（未开始/直播中/已结束）

### P05 战队详情页
- 战队 Logo、名称、所属赛区、简介
- 当前阵容（选手列表，含位置/角色，视数据源而定）
- 近期比赛战绩（近 5-10 场）
- 近期/当前所参加赛事
- 收藏按钮

### P06 选手详情页
- 选手头像、ID、真实姓名（如有）、国籍
- 所属战队
- 角色定位（如决斗者/哨兵/先锋/控场，视数据源）
- 基础数据（如 KDA、ACS 等，取决于数据源是否可得）
- 收藏按钮

### P07 搜索页
- 搜索输入框 + 历史搜索记录（本地存储）
- 结果分类展示：赛事/战队/选手 分组列表

### P08 我的收藏页
- Tab 分类：战队/选手/赛事
- 对应卡片列表，支持取消收藏

### P09 我的页
- 收藏入口快捷方式
- 关于 VTCC
- 意见反馈
- （可选）版本号

## 7. MVP 范围（本期不做）

**明确不做（V1 不涉及）**

- 用户账号体系/登录注册（收藏用本地存储 + 微信 openid 即可，不做完整账号系统）
- 比赛录像/集锦播放
- 实时文字直播/比分推送通知（订阅消息可放 V2）
- 选手/战队详细的历史交锋对比分析
- 社区互动功能（评论、点赞、讨论区）
- 竞猜/预测功能
- 多语言支持（先做中文）
- 深度数据可视化（雷达图、热力图等）
- 赛程日历订阅/导出到微信日历
- 内容资讯/新闻（避免变成"资讯媒体"，聚焦"工具"定位）
- 分享海报生成、裂变功能

> MVP 边界原则：只做「查询」，不做「互动」、不做「社区」、不做「内容」。

## 8. 后续版本规划

### V2（体验与粘性）
- 微信订阅消息：关注战队/选手的比赛开赛提醒
- 首页个性化：根据收藏自动置顶相关比赛
- 历史交锋（H2H）：两支战队/选手对战历史
- 选手/战队数据榜单（KDA 榜、爆头率榜等）
- 分享功能：分享比赛/战队卡片到好友或群

### V3（深度与生态）
- 数据可视化增强（趋势图、赛季数据对比）
- 竞猜/预测互动（非博彩性质，纯娱乐积分）
- 赛事资讯/短内容流（战报、集锦文字总结）
- 多赛区数据横向对比工具
- 开放 API/数据给第三方自媒体使用
- 探索小游戏化激励体系（签到、成就徽章）

## 9. 风险分析

| 风险类型 | 具体风险 | 应对建议 |
|---|---|---|
| 数据源风险 | Valorant/VCT 官方无公开 API，数据获取依赖爬虫或第三方数据源（如 vlr.gg、liquipedia），存在合法性和稳定性风险 | 优先调研是否有官方/半官方数据接口；爬虫需评估目标网站 robots 协议与法律风险；做好数据源异常的降级与缓存机制 |
| 版权风险 | 战队 Logo、选手照片、赛事素材涉及第三方版权（Riot Games 及战队俱乐部） | 咨询法务边界，非商业化展示通常风险较低，但如涉及商业变现需谨慎；必要时使用简化图标替代 |
| 微信小程序合规风险 | 电竞数据类小程序若涉及「赛事竞猜/博彩」倾向易被下架；需确认类目备案（游戏电竞资讯类） | MVP 阶段坚决不做竞猜/博彩相关功能；提前确认小程序类目审核要求 |
| 数据时效性风险 | 比分/赛程更新不及时会严重影响用户信任 | 建立数据更新监控机制，明确数据更新频率（如 5-10 分钟一次），页面标注「数据更新于 XX 分钟前」 |
| 冷启动风险 | 工具类产品无内容驱动，获客成本高，用户复访动机弱 | 前期结合社区（贴吧、NGA、微博电竞话题、B 站）做种子用户导流；核心靠「快查赛程」的刚需场景驱动自然传播 |
| 数据颗粒度风险 | 选手详细数据（ACS、经济数据等）可能没有可靠开放数据源 | MVP 阶段选手数据字段可先做「基础信息 + 战队归属」，数据类字段留空或延后到有稳定数据源时上线 |

## 10. 技术建议（微信原生小程序 + Node.js + MySQL）

### 整体架构

```
微信小程序（原生框架）
      ↓ HTTPS
Node.js 后端（Express/Koa/Nest.js）
      ↓
MySQL 数据库 + Redis 缓存（建议加入）
      ↑
数据采集层（定时任务/爬虫服务）
      ↑
外部数据源（vlr.gg / liquipedia 等）
```

### 前端（小程序）建议
- 原生小程序 + 组件化开发，公共组件抽离：战队卡片、选手卡片、比赛卡片组件复用度高
- 用 `wx.request` 统一封装请求层，做好 loading/error/空状态三态处理
- 收藏功能：MVP 阶段可用 `wx.getStorageSync` 本地存储 + 用户 openid 关联，避免做完整账号体系
- 图片资源（Logo 等）建议接入 CDN，避免直接存储在服务器

### 后端建议
- 框架选型：Node.js + Express 或 Koa（轻量、上手快，适合 MVP）；如团队更看重工程规范，可选 Nest.js
- 分层设计：Controller（接口层）→ Service（业务逻辑）→ DAO（数据访问层），便于后续扩展
- 数据更新：独立的定时任务服务（node-cron），负责定期从数据源拉取/更新赛事、比赛、战队、选手数据，与用户请求接口解耦
- 接口设计：RESTful 风格，核心接口如 `/api/matches/today`、`/api/tournaments/:id`、`/api/teams/:id`、`/api/players/:id`、`/api/search?q=`

### 数据库设计（MySQL）核心表建议

| 表名 | 说明 | 核心字段 |
|---|---|---|
| tournaments | 赛事表 | id, name, region, stage, start_time, end_time, logo_url, status |
| matches | 比赛表 | id, tournament_id, team_a_id, team_b_id, match_time, bo_type, status, score_a, score_b |
| match_maps | 分图比分表 | id, match_id, map_name, score_a, score_b, order |
| teams | 战队表 | id, name, region, logo_url, description |
| players | 选手表 | id, team_id, nickname, real_name, nationality, role, avatar_url |
| favorites | 收藏表 | id, openid, target_type(team/player/tournament), target_id, created_at |

### 缓存建议
- 强烈建议引入 Redis：今日比赛、热门赛事等高频只读数据做缓存，减轻 MySQL 压力，同时缓解数据源抓取延迟对用户的影响
- 缓存策略：比赛进行中数据缓存时间短（1-2 分钟），已结束比赛数据可长期缓存

### 其他建议
- 日志与监控：数据采集任务需有失败告警机制（如钉钉/企业微信机器人通知），避免数据源接口变化导致数据停更却无人知晓
- 接口版本化：预留 `/api/v1/` 前缀，方便 V2 迭代时的接口演进
- 搜索功能 MVP 阶段可用 MySQL 的 `LIKE` 模糊查询实现，数据量增长后再考虑 Elasticsearch
